   124→## Sección 14: Seguridad para Automatizaciones e Inteligencia Artificial
   125→> **Tagline:** *La automatización no protegida es una bomba de tiempo. La IA sin seguridad es una puerta trasera.*  
   126→> **Nivel:** Competente | **Horas estimadas:** 10h | **Icono:** 🔒
   127→
   128→### Relevancia laboral
   129→La seguridad en automatizaciones e IA es el gap más grande del mercado peruano y LATAM. Las empresas US y globales requieren que sus ingenieros de AI/Automation entiendan el OWASP LLM Top 10, protejan pipelines de datos, y defiendan modelos contra ataques de prompt injection. En 2026, el 65% de las brechas de seguridad involucran algún componente de IA o automatización. Esta sección es obligatoria para cualquier perfil mid-senior.
   130→
   131→### Objetivos de aprendizaje
   132→- Entender y aplicar el OWASP LLM Top 10 (2025) a aplicaciones reales con IA
   133→- Implementar defensa en capas contra prompt injection directa e indirecta
   134→- Usar `presidio` de Microsoft para anonimización de PII en pipelines de datos
   135→- Aplicar `detect-secrets` y gestión segura de credenciales con variables de entorno
   136→- Entender hashing y criptografía con `hashlib`, `cryptography` (cubierto en Python 201 Ch. 14)
   137→- Implementar logging de auditoría y trazabilidad en bots y agentes
   138→- Aplicar el principio de mínimo privilegio en automatizaciones con APIs y bases de datos
   139→
   140→### Temas de teoría
   141→1. **OWASP LLM Top 10 (2025):** LLM01 Prompt Injection, LLM02 Sensitive Info Disclosure, LLM03 Supply Chain, LLM04 Data Poisoning, LLM05 Improper Output Handling, LLM06 Excessive Agency, LLM07 System Prompt Leakage, LLM08 Vector/Embedding Weaknesses, LLM09 Misinformation, LLM10 Unbounded Consumption
   142→2. **Criptografía práctica en Python** (Python 201, Ch.14): `hashlib` para SHA-256 hashing, PBKDF2 para key derivation, `cryptography` con Fernet (symmetric) y RSA (asymmetric)
   143→3. **Gestión de secretos:** `python-decouple`, `python-dotenv`, AWS Secrets Manager básico, nunca hardcodear API keys
   144→4. **Anonimización y PII:** Microsoft `presidio` — detección y anonimización de NIT, DNI, correos, teléfonos peruanos en DataFrames
   145→5. **Defensa contra prompt injection en 6 capas:** (1) input validation con regex, (2) system prompt restrictivo, (3) output filtering, (4) privilege control en tools, (5) human-in-the-loop para acciones críticas, (6) adversarial testing
   146→6. **`detect-secrets` como pre-commit hook:** detectar API keys, passwords y tokens antes de hacer git commit
   147→7. **Logging de auditoría con `structlog`:** logs inmutables, estructurados (JSON), trazables, con campos: timestamp, user_id, input_hash, output_hash, latencia, security_flags
   148→8. **Seguridad en automatizaciones RPA:** sanitizar inputs web, rotar credenciales, usar storage_state de Playwright con cifrado
   149→
   150→### I Do — Demostración guiada (Yo hago)
   151→El instructor construye paso a paso con explicación del por qué:
   152→1. Una función `sanitize_prompt(user_input: str) -> str` que limpia entradas con regex y logging de intentos sospechosos
   153→2. Un script con `presidio-analyzer` + `presidio-anonymizer` para anonimizar DNIs y correos en un DataFrame de pandas
   154→3. Un pipeline de verificación de integridad con `hashlib` SHA-256 para archivos de modelos ML
   155→4. Configuración de `detect-secrets` como pre-commit hook en el repo del curso con `git commit` que falla si detecta un secret

   157→### We Do — Práctica guiada (Hacemos juntos)
   158→El estudiante escribe código con guía, starter code, hints y solución:
   159→1. Implementar un `PromptGuard` class con: validación de longitud máxima, detección de patrones de injection via regex, y `structlog` logging de cada violación
   160→2. Construir un `AuditLogger` con `structlog` que registra: timestamp, user_id, input hash (SHA-256), output hash, latencia en ms, security flags detectados
   161→3. Simular los 3 ataques más comunes del OWASP LLM Top 10 en un chatbot de juguete (prompt injection, system prompt leakage, excessive agency) y aplicar las mitigaciones correctas

   190→## Sección 15: Python Avanzado — stdlib Profunda
   191→> **Tagline:** *El Python que todos usan superficialmente. Tú lo dominarás en profundidad.*  
   192→> **Nivel:** Competente | **Horas estimadas:** 12h | **Icono:** ⚙️
   193→
   194→### Relevancia laboral
   195→Las entrevistas técnicas de empresas US para roles Python Senior testean: context managers, functools, itertools, descriptors y type hints. Esta sección cubre íntegramente los capítulos de Python 201 que no tenían sección dedicada en la versión original del roadmap: Ch. 3 (contextlib), Ch. 4 (functools), Ch. 8 (itertools), Ch. 10 (typing), Ch. 12 (Unicode), Ch. 16-18 (super, descriptors, scope).
   196→
   197→### Objetivos de aprendizaje
   198→- Crear context managers con `contextlib`: `contextmanager`, `closing`, `suppress`, `ExitStack`
   199→- Usar `functools` avanzado: `lru_cache`, `partial`, `singledispatch`, `wraps`, `reduce`
   200→- Explotar `itertools`: infinite iterators (`count`, `cycle`, `repeat`), combinatorics (`product`, `combinations`, `permutations`), terminators (`groupby`, `islice`, `chain`)
   201→- Implementar descriptors con `__get__`, `__set__`, `__delete__` para validación de atributos
   202→- Aplicar type hints completos con `typing`: `Protocol`, `TypeVar`, `Generic`, `Literal`, `TypedDict`, `overload`
   203→- Manejar Unicode y encoding correctamente en pipelines de datos con texto peruano (ñ, tildes, caracteres especiales)
   204→- Comprender MRO y `super()` en herencia múltiple (diamond problem)
   205→- Dominar scope: closures, `nonlocal`, fábricas de funciones
   206→
   207→### Temas de teoría
   208→1. **Context Managers** (Python 201, Ch. 3): `with` statement internals, `__enter__`/`__exit__`, `contextlib.contextmanager`, `contextlib.ExitStack` para múltiples CMs dinámicos
   209→2. **functools avanzado** (Python 201, Ch. 4): `lru_cache(maxsize=128)` para memoización, `partial` para fijar argumentos (currying), `singledispatch` para function overloading por tipo, `wraps` para decorators bien formados que preservan `__name__` y docstring
   210→3. **itertools arsenal** (Python 201, Ch. 8): `count(start, step)`, `cycle(iterable)`, `repeat(obj, times)`, `chain(*iterables)`, `islice(iterable, stop)`, `groupby(iterable, key)`, `product(*iterables, repeat)`, `combinations(iterable, r)`, `permutations(iterable, r)`
   211→4. **Descriptors** (Python 201, Ch. 17): non-data vs data descriptors, `__set_name__`, uso en frameworks (Django fields, SQLAlchemy columns)
   212→5. **Type Hints modernos** (Python 201, Ch. 10): `Protocol` para duck typing estructurado, `TypeVar` para genéricos, `Generic[T]`, `Literal["value"]`, `TypedDict`, `@overload` para múltiples firmas
   213→6. **Unicode en profundidad** (Python 201, Ch. 12): code points, UTF-8 vs UTF-16, `str.encode()`, `bytes.decode()`, `chardet` para detectar encoding de archivos, problemas comunes con CSVs de sistemas legacy peruanos
   214→7. **super() y MRO** (Python 201, Ch. 16): Algoritmo C3 linearization, `type.__mro__`, diamond problem resuelto
   215→8. **Scope y closures** (Python 201, Ch. 18): LEGB rule, `nonlocal`, cell objects, factories de funciones para currying manual
   216→

   505→### Temas de teoría

   512→7. **HyDE:** generar un documento hipotético con el LLM y buscar embeddings similares — mejora recall en preguntas complejas

   611→## Sección 22: RapidFuzz y Resolución de Entidades a Escala
   612→> **Tagline:** *Los datos del mundo real están sucios. Esta sección te enseña a reconciliarlos como un profesional.*  
   613→> **Nivel:** Competente | **Horas estimadas:** 10h | **Icono:** 🔎

   615→### Relevancia laboral
   616→Entity resolution (ER) y fuzzy matching son técnicas críticas en Data Engineering, Master Data Management y proyectos de integración. El proyecto "Familiarity Score Dashboard" del curso lo requiere explícitamente. `RapidFuzz` es hasta 20x más rápido que `fuzzywuzzy` y es el estándar actual. En proyectos de linkage de datos peruanos (DNI, nombres, direcciones), estas técnicas son indispensables.

   619→- Usar `RapidFuzz` con las métricas correctas: `ratio`, `partial_ratio`, `token_sort_ratio`, `WRatio`, `jaro_winkler`
   620→- Implementar el pipeline de 4 pasos: normalizar → bloquear → matchear → auditar
   621→- Diseñar estrategias de blocking por campo de alta cardinalidad para escalar a O(n log n)
   622→- Entender cuándo `RapidFuzz` es suficiente vs cuándo escalar a `Splink` o `dedupe.io`
   623→- Aplicar entity resolution a datos peruanos: nombres, DNIs, distritos, empresas

   627→### Temas de teoría
   628→1. **Distancias de strings:** Levenshtein, Jaro-Winkler, Coseno, Hamming — cuándo usar cada una
   629→2. **RapidFuzz API:** `fuzz.ratio`, `fuzz.partial_ratio`, `fuzz.token_sort_ratio`, `process.extract`, `process.cdist`
   630→3. **Pipeline de 4 pasos** (patrón de producción):
   631→   - Paso 1 — Normalizar: mayúsculas, stripear espacios, quitar tildes, colapsar variantes (Phase II → phase 2)
   632→   - Paso 2 — Bloquear: particionar por campo de alta cardinalidad antes de comparar (escala O(n²) → O(n))
   633→   - Paso 3 — Matchear: aplicar RapidFuzz dentro de cada bloque con umbral configurable
   634→   - Paso 4 — Auditar: loggear cada merge con keys originales, score, regla y timestamp (reversible)
   635→4. **Fuzzy matching de DNIs y nombres peruanos:** variantes de nombres hispanos, abreviaciones, transliteraciones
   636→5. **Umbral óptimo:** cómo calibrar el threshold usando un conjunto de pares etiquetados manualmente
   637→6. **Escalando a Splink:** cuándo los señales de "blocking inestable", "ground truth contestado" o "schema churn" indican que necesitas más potencia
   638→7. **Integración con pandas:** `process.cdist()` para matrices de similaridad vectorizadas
