Protocolo de reanudación: (1) valida schemas y hashes del checkpoint; (2) confirma que la última revisión buena existe; (3) compara ledger, issues y artefactos con ese checkpoint; (4) repite el último test registrado; (5) reanuda únicamente `next_single_action`. ## Perfil recomendado de ejecución

* Modelo: `glm-5.2`.
* Tareas de investigación, arquitectura, debugging y revisión global: `thinking={"type":"enabled"}` y `reasoning_effort="max"`.
* Autoría de un subtema ya diseñado: `reasoning_effort="high"` como punto de partida.
* Formato, renombre o edición mecánica: thinking puede desactivarse por turno.
* Sampling: no ajustar `temperature` y `top_p` simultáneamente. Crear evals antes de elegir; para documentación técnica estable, probar `top_p=0.8` contra defaults.
* Para una vía de menor variación, comparar un perfil muestreado contra `do_sample=false`; no prometer reproducibilidad bit a bit y, cuando `do_sample=false`, no atribuir efecto a `temperature` o `top_p`.

* Herramientas: validar argumentos antes de ejecutar. Con streaming, concatenar argumentos y comprobar `finish_reason`.

* Para selección automática usa `tool_choice="auto"` según la API vigente. Antes de ejecutar valida permisos, side effects, idempotencia y alcance. Conserva el orden de `tool_calls`; devuelve cada resultado con `role="tool"` y el `tool_call_id` exacto, sin reordenar mensajes assistant/tool.
* `finish_reason="tool_calls"` es un estado intermedio: valida y ejecuta todas las llamadas, incorpora sus resultados y continúa. Solo `stop`, sin herramientas ni acciones pendientes y con el gate verificado, puede cerrar una fase. `length`, `network_error`, `model_context_window_exceeded` o `sensitive` nunca cierran una fase; tampoco la cierran un error HTTP, JSON inválido, stream truncado o ausencia de `finish_reason`.
* Si se habilita preserved thinking, reenviar `reasoning_content` exactamente como fue recibido y usar `thinking={"type":"enabled","clear_thinking":false}`. Si el cliente no puede garantizarlo, usar `thinking={"type":"enabled","clear_thinking":true}` y un checkpoint externo.

  "misconceptions_targeted": ["..."],
  "expected_minutes": 4,
  "difficulty_target": "...",
  "environment": "browser-pyodide|local-python|cloud-lab",
  "starter_or_fixture": "...",
  "tests_or_assertions": ["..."],
  "accessibility_notes": ["..."],
  "sources": ["https://..."]
}
```

Para `mcq`, `options` contiene exactamente las opciones mostradas, hay una sola mejor respuesta y existe racional para cada distractor mostrado. Para `trace|debug|complete|write_code`, `options=null`, `distractor_rationales=null`, `correct_answer` es una clave/rúbrica verificable y `tests_or_assertions` decide el puntaje. `variant_id` es obligatorio y único; `expected_minutes` es mayor que cero. No fuerces código auténtico a formato MCQ.

Resuelve y ejecuta cada variante de forma independiente. Compara A/B/C en habilidad, formato, número de decisiones, demanda cognitiva, entorno, tiempo y dificultad prevista. Balancea posiciones de clave a escala de sección y revisa que cada distractor corresponda a una concepción errónea plausible. Si divergen, reescribe; no promedies la discrepancia.

Gate: existen exactamente 24 variantes —tres por cada uno de ocho subtemas—, IDs únicos, ninguna familia mezcla outcomes o formatos, cada opción mostrada coincide con su clave/racional, todas las claves/tests pasan y la revisión de equivalencia no deja P0/P1 abiertos.

### PHASE 6 — Validación aislada

El validador recibe solo el contenido que verá el estudiante y los enunciados, no las soluciones.

Debe:

* resolver ejercicios y examen;
* citar la parte del contenido que permite resolverlos;
* ejecutar código en el entorno declarado;
* reportar ambigüedad, conocimiento no enseñado, hints insuficientes, distractores débiles y tests faltantes;
* revisar privacidad, accesibilidad y lenguaje.

El supervisor compara contra claves y rubricas sin revelar soluciones al validador durante el intento.

Gate: toda falla genera issue; ninguna se corrige solo cambiando la prueba para que pase.

Gate: el validador aislado completa el paquete sin soluciones filtradas; contenido, ejercicios, cuatro evaluaciones, banco, examen, incremento de proyecto, privacidad y accesibilidad tienen evidencia. Toda ambigüedad o dependencia no enseñada queda reparada o bloquea el cierre.

Enrutamiento: si todo pasa, continúa a PHASE 8; si algo falla, continúa a PHASE 7. Nunca saltes directamente de una validación fallida al cierre.

### PHASE 7 — Root cause, reparación y regresión

Para cada falla:

1. Reproduce.
2. Identifica causa, no solo síntoma.
3. Evalúa alcance e interacciones.
4. Implementa el cambio mínimo completo.
5. Ejecuta test específico.
6. Ejecuta regresión de sección y flujo.
7. Actualiza issue y matriz de requisitos.

Máximo tres ciclos sin progreso sobre la misma hipótesis. Después, cambia de hipótesis o marca bloqueo con evidencia.

Gate: cada issue reparado conserva reproducción previa, cambio aplicado y regresión posterior; cero P0/P1 abiertos. Un issue `blocked` P0/P1 impide `passed` aunque el contenido documental esté completo.

Enrutamiento: después de cualquier reparación vuelve obligatoriamente a PHASE 6. Solo una PHASE 6 completamente aprobada llega a PHASE 8; una reparación bloqueada conserva `blocked` y cierra sin `passed`.

### PHASE 8 — Cierre y checkpoint

Guarda:

```json
{
  "run_id": "...",
  "prompt_version": "...",
  "model": "glm-5.2",
  "model_parameters": {},

  "input_hashes": {},
  "artifact_hashes": {},

  "scope_completed": ["..."],
  "requirements_status": {"REQ-ID": "pass|fail|blocked"},
  "decisions": ["..."],
  "files_changed": ["..."],

  "trace_ids": [],
  "tool_failures": [],
  "finish_reason": "stop",
  "cached_tokens": 0,
  "open_issues": ["..."],
  "sources": ["..."],
  "next_single_action": "...",
  "resume_preconditions": []
}
```

Resume el resultado, limitaciones y siguiente acción. Mantén un ledger con una fila por sección y estados `not_started|researching|authoring|validating|repairing|passed|blocked`; “reparado” es evidencia de un issue, no un estado terminal de sección.

La instrucción persistente “procesar S1–S52” autoriza la transición automática de `Sx=passed` a `S(x+1)=researching` solo cuando `x < 52`; no autoriza saltar gates, trabajar secciones en paralelo ni avanzar desde `blocked`. En S4/S8/S13, S17/S21/S26, S30/S34/S39 y S43/S47/S51 ejecuta el gate del capstone correspondiente. En S13/S26/S39/S51 ejecuta además la regresión de nivel. Para `x = 52`, no crees S53: ejecuta exclusivamente el gate del capstone FINAL —integración de los doce, regresión completa, demo reproducible, documentación de publicación, rúbrica aprobada y cero fallas críticas— y deriva `course_complete`.

`course_complete` se deriva y solo puede ser `true` cuando 52/52 secciones, 12/12 capstones de nivel y el capstone FINAL están `passed`, y no existe P0/P1 abierto. No existe S53 ni un gate implícito que permita compensar el fallo de un capstone con la nota de otro.

Antes de iniciar la sección siguiente, abre un contexto nuevo con solo: contrato, fila del roadmap, checkpoint previo, decisiones globales, issues abiertos y paths necesarios. No arrastres texto irrelevante de secciones cerradas.

Protocolo de reanudación: (1) valida schemas y hashes del checkpoint; (2) confirma que la última revisión buena existe; (3) compara ledger, issues y artefactos con ese checkpoint; (4) repite el último test registrado; (5) reanuda únicamente `next_single_action`. Si falla cualquier paso, cambia la sección a `blocked`, conserva artefactos y diagnostica; nunca reconstruyas el estado desde memoria ni marques una fase cerrada por el texto de una conversación.

Gate de cierre: `finish_reason="stop"`, ninguna herramienta/acción pendiente, artefactos y hashes guardados, tests repetibles, cero P0/P1, DoD completa y ledger/checkpoint coherentes. Solo entonces escribe `passed`.

## Evals del propio prompt

Antes de adoptar este prompt compara la versión anterior y la nueva sobre una muestra estratificada y retenida:

1. Dos secciones del Nivel 1, incluida una integración de S13.
2. Dos secciones del Nivel 2, incluida una integración de S26.
3. Dos secciones del Nivel 3, incluida una integración de S39.
4. Dos secciones del Nivel 4, incluida una integración de S52.
5. Un issue real de plataforma.

Métricas:

* cobertura de REQ-ID;
* exactitud técnica revisada;
* código que pasa tests;
* completitud estructural;
* equivalencia A/B/C por expertos y luego por piloto;
* calidad de distractores;
* trazabilidad de fuentes;
* issues detectados antes de implementación;
* costo, latencia, truncamientos y `finish_reason`;
* acuerdo del evaluador con humanos.
* precisión de selección de herramientas y argumentos;
* cumplimiento de gates y recuperación desde checkpoint;
* robustez cuando una restricción crítica aparece al inicio, centro o final de contexto largo;
* tokens cacheados y tamaño del contexto activo.

Usa conjunto de desarrollo y conjunto retenido. No optimices el prompt contra todos los casos. En LLM-as-judge, usa rúbrica por criterio, invierte orden de candidatos y calibra con revisión humana.
